# Support operations

Koru has no supported release. Before public beta, maintainers must assign owners and enable the GitHub features referenced here.

## Channels

- Discussions: questions, ideas, and general compatibility help using synthetic examples.
- Issues: reproducible bugs and approved work; forms reject private content by default.
- Private vulnerability reporting: suspected security/privacy vulnerabilities or exposure.

GitHub Discussions, labels, branch protection, and private vulnerability reporting are external manual gates and are not configured by repository files alone.

## Triage

1. Remove/privatize exposed content immediately; never quote it into another issue.
2. Classify severity using the quality plan. Critical/High blocks release.
3. Capture app/build, macOS/architecture, permission enums, capability label, reason code, and synthetic reproduction.
4. Route permission, Accessibility, event tap, pasteboard, insertion, migration/integrity, or key-loss state using the previewed support bundle.
5. Link a regression test/evidence and update compatibility/troubleshooting/release notes.

No maintainer should request a user's vault, Keychain material, raw unified logs, private screenshot, real clipboard entry, saved item, typed query, file path, URL, or document/window name in public support.
